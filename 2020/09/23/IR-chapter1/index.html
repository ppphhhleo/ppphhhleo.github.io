Chapter 1 Boolean Retrieval

Brief introduction:

Chapter 1 introduces inverted indexes and shows how to build an inverted index, and it demonstrates how simple Boolean queries can be processed using such indexes with algorithms shown. In this chapter, we begin with a very simple example of an IR problem, and introduce the idea of a term-document matrix (Section 1.1) and the central inverted index data structure (Section 1.2). We then examine the Boolean retrieval model, how Boolean queries are processed and general query optimization (Sections 1.3). We discuss both advantages and shortcomings of Boolean retrieval at practical level.

IR

Information retrieval (IR) is finding material (usually documents) of an unstructured nature (usually text) that satisfies an information need from within large collections (usually stored on computers).

Information retrieval can cover various kinds of data and information, such as unstructured data, latent linguistic structure, and semi structured data. Information retrieval systems can also be distinguished by three scales where they operate, which are web search, personal information retrieval, and the space of enterprise, institutional, and domain-specific search.

The field of IR also supports users browsing or filtering documents, or further processing retrieved documents.

1.1 An example information retrieval problem

Suppose we wanted to find which plays of Shakespeare contain words Brutus AND Caesar AND but NOT Calpurnia, and we also want ranked retrieval (the best answer to return).

Grepping through text could be effective to find certain words, but it’s slow when processing large corpora. NOT Calpurnia is non-trivial with Unix command grep, and other operations (e.g., find the word Romans near countrymen) are not feasible with only grep.

Therefore, indexing the document in advance can process a binary term-document incidence matrix with terms as indexed units. 1 in the matrix represents document containing the term, 0 otherwise, then we have 0/1 vector for each term, with the operators AND, OR, and NOT to answer queries.

However, if the documents to be retrieved are larger collections (e.g. 1 million documents), the term-document matrix could be extremely sparse with few nonzero entries.

e.g.
A 500K ×1M matrix has half-a-trillion 0’s and 1’s – too many to fit in a computer’s memory. Because each document is 1,000 words long, the matrix has no more than one billion 1’s, so a minimum of 99.8% of the cells are zero.

A much better representation is to record only the terms that do occur, which is the idea central to inverted index in information retrieval. For each term k, we must store a list of all documents that contain k with docID (successive integers) to identify each document. Thus we keep a dictionary of terms and variable-size postings lists in case of modification of the documents, and the lists are sorted by docID.

1.2 A first take at building an inverted index

The major steps in building the index: first, collect all documents needed to be indexed; next, tokenize the documents; then, do linguistic preprocessing, producing modified and normalized tokens as the input to indexing; index the documents that each term occurs in by creating an inverted index, consisting of a dictionary and postings.
The core indexing step is sorting alphabetically the list of pairs of terms and docID, and the postings are secondarily sorted by docID. In the resulting index, we merge multiple term entries in different single document, and split the list into dictionary and postings, kept in memory and on disk respectively. Summary information such as, document frequency (the length of each postings list) is also record and added in the dictionary.
For an in-memory postings list, two good alternatives are singly linked lists or variable length arrays. Singly linked lists allow cheap insertion of documents into postings lists and naturally extend to more advanced indexing strategies such as skip lists (later section), requiring additional pointers. Variable length arrays win in space requirements by avoiding pointers and in time requirements because their use of contiguous memory increases speed on modern processors with memory caches.

1.3 Processing Boolean queries

Consider processing a simple conjunctive query using an inverted index and the basic Boolean retrieval model:

Brutus AND Calpurnia.

We need to locate both Brutus and Calpurnia in the dictionary to retrieve both postings, then we need to efficiently intersect the document sets so as to find documents containing both terms. Using the merge algorithm is one simple and effective method of intersecting postings lists: We maintain pointers into both lists and walk through the two postings simultaneously, in time linear in the total number of posting entries. If the lengths of the postings lists are x and y, the intersection takes O (x + y) operations.

Merge algorithm for the intersection of two postings list p1 and p2: 

Intersect (p1, p2)
1 answer ← ()
2 while p1 ≠ NIL and p2 ≠ NIL
3 do if docID(p1) = docID(p2)
4 then Add (answer, docID(p1))
5 p1← next(p1)
6 p2← next(p2)
7 else if docID(p1) < docID(p2)
8 then p1← next(p1)
9 else p2← next(p2)
10 return answer

Knowing how to ‘merge’ two postings lists, we can extend the intersection operation to process more complicated queries, such as a query with an AND of n terms. For each of the n terms, we get its postings, then AND them together, but

what is the best order for query processing (‘merging’ all postings)?

We can accomplish the query optimization with a standard heuristic to process terms in order of increasing document frequency (stored in the dictionary). If we intersect the two smallest postings lists initially, then all intermediate results must be no bigger than the smallest postings list, and we could do the least amount of total work.
Besides, as for a query with OR first then AND of n terms, such as (a OR b) AND (c OR d) AND (e OR f) … we get document frequency for all terms, and then we can implement query optimization by estimating the size of each OR by the sum of its document frequency, then processing in increasing order of OR sizes.

1.4 The extended Boolean model versus ranked retrieval

The Boolean retrieval model is being able to ask a query that is a Boolean expression, and the Boolean queries are queries using AND, OR, NOT to join query terms.

Boolean retrieval were primary commercial retrieval tools for three decades. Westlaw provides commercial Boolean searching service, and here is an example query:

Example query:

Requirements for disabled people to be able to access a workplace

Query:

disabl! /p access! /s work-site work-place employment /3 place

symbols:

disabl! = words starting with disabl
/p = in same paragraph
/s = in same sentence
/3 = within 3 words
Work-site = worksite, work site…

Boolean queries are precise with the use of proximity operators, uncommon in web search. Typical expert queries are usually carefully defined and incrementally developed, and many professional searchers still like Boolean search for greater control and transparency over what is retrieved. However, this does not mean that Boolean queries work actually better and most people seldom use Boolean queries. Turtle (1994) found that free text queries produced better results than Boolean queries.A general problem with Boolean search is that using AND operators tends to produce high precision but low recall searches, while OR operators just the reverse, and it is difficult or impossible to find a satisfactory middle ground.
